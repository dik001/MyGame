# Полная документация проекта The History of the Rune

Версия документа: по состоянию кода `Alpha_v.0.8` в папке `ONLY_CODE/game-rewrtie`.

Этот файл задуман как единая подробная сводка по проекту: что уже реализовано, как связаны системы, где лежит логика, как работает основной игровой цикл, какие механики влияют друг на друга и какие численные значения сейчас зашиты в коде.

## 1. Что это за проект

`The History of the Rune` на текущем этапе представляет собой 2D narrative/life-sim/survival проект на Godot, где игрок управляет Руной и должен прожить ближайшие дни, заработать деньги и заплатить аренду. В проекте уже соединены:

- бытовое выживание: голод, энергия, гигиена, здоровье;
- психологическое давление: настроение и стресс;
- экономика: наличные, банковский счет, аренда, покупки;
- работа: фриланс-заказы и мини-игра модерации;
- домашний быт: сон, ванна, холодильник, компьютер;
- внешние взаимодействия: магазин, банкомат, телефон, сообщения, аренда через LeChat;
- отдельная мини-игра `Slithario` как дополнительный источник наличных.

Главная драматическая ось проекта уже заложена в прологе: у Руны мало времени, нужно продержаться, организовать быт и найти деньги на аренду.

## 2. Техническая база

- Движок: `Godot 4.6`.
- Главная сцена проекта: `res://scenes/main/title_screen.tscn`.
- Основная игровая сцена: `res://scenes/main/game.tscn`.
- Базовая игровая комната по умолчанию: `res://scenes/rooms/apartament.tscn`.
- Базовое разрешение viewport: `1920x1080`.
- Проект активно использует `autoload`-синглтоны для глобального состояния.
- В проекте есть явные исторические опечатки в именах файлов и папок, например:
  - `game-rewrtie`;
  - `apartament.tscn`;
  - `enterance.tscn`.

Это не ломает логику, но важно помнить при навигации по репозиторию.

## 4. Состояние новой игры

При старте новой игры проект ведет игрока через следующий набор стартовых условий:

- запуск идет с `title_screen.tscn`;
- затем загружается `game.tscn`;
- активная комната по умолчанию: квартира `apartament.tscn`;
- время на старте: `день 1, 07:25`;
- стартовые деньги:
  - наличные: `$25`;
  - банк: `$0`;
- стартовый инвентарь содержит стартовый школьный комплект:
  - `school_top`;
  - `school_bottom`;
  - `school_shoes`;
- этот комплект автоматически экипируется, если найден в инвентаре;
- холодильник может инициализироваться со стартовым яблоком;
- базовая цель до завершения пролога: `Найти деньги на аренду`;
- при первом заходе в квартиру запускается сюжетный пролог Руны;
- после пролога цель обновляется на `Прожить 7 дней и оплатить аренду`.

## 5. Управление

Кастомные действия из `project.godot`:

- `move_left`: `A`
- `move_right`: `D`
- `move_up`: `W`
- `move_down`: `S`
- `interact`: `E`
- `open_inventory`: `I`
- `open_phone`: `Up`
- `pause_menu`: `Escape`

Дополнительно:

- диалоги продолжаются через кнопку взаимодействия, Enter или клик;
- в мини-игре модерации:
  - `A` или `Left` = одобрить;
  - `D` или `Right` = отклонить;
- `Escape` в desktop:
  - закрывает текущее fullscreen-приложение;
  - если приложений не открыто, возвращает из desktop обратно в `game.tscn`;
- `F3` в debug-сборках включает отладочный overlay `WorldGrid`.

### Как сцены связаны между собой

Высокоуровневая схема такая:

`title_screen.tscn` -> `game.tscn` -> активная комната мира -> взаимодействия -> desktop/мини-игры/окна -> возврат обратно в `game.tscn` c восстановлением комнаты и позиции игрока.

`GameManager` хранит:

- путь текущей комнаты;
- целевой `portal_id` для появления;
- снимок runtime-мира на момент выхода из основной сцены;
- механизм `queue_runtime_world_restore()` для безопасного возврата из desktop и мини-игр.

## 7. Карта мира и комнаты

На текущем этапе в проекте пять основных world-сцен.

### 7.1. `apartament.tscn`

Квартира игрока. Здесь находятся:

- кровать;
- компьютерный стол;
- холодильник;
- ванна;
- дверь выхода в подъезд;
- стартовая точка появления `SceneStartPoint`.

Это самая важная безопасная сцена проекта. Для ментальной системы квартира считается домашней безопасной зоной.

### 7.2. `enterance.tscn`

Подъезд/лестничная площадка между квартирой и лифтом.

Внутри есть:

- дверь обратно в квартиру;
- дверь/переход к сцене лифта.

### 7.3. `elevator.tscn`

Отдельная сцена лифта. В ней:

- нет обычного room-геймплея;
- игроку блокируют свободное перемещение;
- основной интерактивный объект: `ElevatorControlPanel`.

Панель предлагает две точки назначения:

- квартирный этаж: `enterance.tscn`, портал `hall_elevator`;
- улица: `town.tscn`, портал `town_building`.

### 7.4. `town.tscn`

Городская улица. Здесь находятся:

- банкомат;
- вход обратно в дом/лифт;
- вход в супермаркет.

### 7.5. `supermarket.tscn`

Магазин. Внутри:

- кассир;
- выход обратно на улицу.

### 7.6. Фактический граф переходов мира

Реальная структура переходов по сценам сейчас выглядит так:

- `apartament.tscn` <-> `enterance.tscn`
- `enterance.tscn` <-> `elevator.tscn`
- `elevator.tscn` <-> `town.tscn`
- `town.tscn` <-> `supermarket.tscn`

Телефонная карта хранит упрощенный логический граф локаций, но для фактического перемещения по миру важны именно сцены и порталы выше.

## 8. Игрок и перемещение

Основная логика игрока реализована в `scenes/Characters/player.gd`.

### 8.1. Базовые принципы

- перемещение клеточное, через `WorldGrid`;
- размер клетки: `32x32`;
- базовая длительность шага: `0.14`;
- задержка повторного шага:
  - первая: `0.18`;
  - последующая: `0.10`;
- диагональное перемещение по умолчанию выключено;
- базовая скорость персонажа в экспорте: `180`;
- расстояние точки взаимодействия: `28`.

### 8.2. Блокировки управления

У игрока есть два разных режима ограничения:

- `input lock` — полная блокировка ввода;
- `movement lock` — частичная блокировка передвижения.

Они используются:

- во время диалогов;
- при открытии модальных окон;
- во время сна;
- во время ванны;
- при лифтовой сцене;
- при открытом хранении, банкомате, desktop-приложениях и других модальных окнах.

### 8.3. Взаимодействия с миром

`WorldGrid` умеет находить интерактивные объекты не только "в клетке перед игроком", а по паттернам:

- `current`
- `front_1`
- `current_or_front`
- `wide_front`
- `line_2`
- `line_3`

По умолчанию для игрока используется `current_or_front`.

Система также учитывает:

- коллизии тайлмапов;
- `PhysicsBody`;
- клетки, занятые интерактивными объектами.

### 8.4. Сохранение состояния игрока

При сохранении runtime-снимка у игрока сохраняются:

- позиция;
- направление взгляда;
- активная grid-cell, если она определена.

## 9. Время, дни и течение мира

`GameTime` управляет глобальным временем.

### 9.1. Базовые правила

- старт игры: `день 1, 07:25`;
- `1 реальная секунда = 1 игровая минута`;
- время хранится как абсолютное число минут;
- есть сигналы:
  - `time_changed`;
  - `day_changed`;
  - `clock_paused_changed`.

### 9.2. Пауза часов

Многие окна и модальные UI-экраны ставят часы на паузу:

- ванна;
- сон;
- хранение;
- магазин;
- банкомат;
- часть инвентарного/диалогового UI.

### 9.3. Отслеживание сна

Система отдельно хранит:

- день последнего начала сна;
- флаг "спал ли игрок сегодня".

Это нужно, чтобы кровать не позволяла повторно спать в тот же игровой день.

## 10. Базовые жизненные показатели

Вся жизненная модель сконцентрирована в `PlayerStats`.

### 10.1. Базовые максимумы

| Параметр | Значение |
| --- | --- |
| HP | `100` |
| Hunger | `100` |
| Energy | `100.0` |
| Hygiene | `100` |

### 10.2. Что именно хранит система

`PlayerStats` отвечает сразу за несколько подсистем:

- здоровье;
- голод;
- энергию;
- гигиену;
- накопленное время бодрствования;
- предупреждения о голоде, сне и грязи;
- запрос принудительного "blackout sleep";
- синхронизацию условий в `FreelanceState`.

## 11. Голод

### 11.1. Пассивное падение голода

- во время бодрствования: `4` единицы в час;
- при сне: еда тоже тратится через sleep-расчет, у сна базовый `hunger_per_hour = -2`.

### 11.2. Пороги состояний голода

| Стадия | Порог |
| --- | --- |
| Hungry | `<= 60` |
| Very hungry | `<= 30` |
| Exhausted by hunger | `<= 10` |
| Starving | `<= 5` |

### 11.3. Эффекты голода

`Very hungry`:

- множитель восстановления энергии: `0.9`;
- множитель скорости движения: `0.95`.

`Exhausted`:

- множитель восстановления энергии: `0.8`;
- множитель скорости движения: `0.9`;
- состояние помечено как запрещающее пассивное восстановление HP.

`Starving`:

- сохраняет штрафы exhausted;
- начинает наносить урон по HP;
- тик урона: `1 HP` каждые `15` реальных секунд.

### 11.4. Еда

Еда восстанавливает голод через `consume_food_hunger()`, а затем одновременно отправляет событие в ментальную систему. Чем заметнее еда закрыла голод, тем сильнее может быть положительный mental impact.

## 12. Энергия, усталость и сон

### 12.1. Базовая модель усталости

Система сна не считает только "полоску энергии". Она еще отслеживает продолжительность бодрствования.

Пороговые стадии:

| Стадия | Время без сна |
| --- | --- |
| Tired | `24` часа |
| Very tired | `36` часов |
| Critical | `44` часа |
| Forced blackout | `48` часов |

Критический порог энергии для сигнала: `15`.

### 12.2. Обычный сон через кровать

Кровать:

- позволяет спать до `06:00`;
- не дает спать повторно в тот же день;
- рассчитывает:
  - сколько минут проспит игрок;
  - сколько восстановится энергии;
  - сколько восстановится HP;
  - сколько снизится голод;
- использует `SleepMath`, где восстановление энергии зависит от текущего HP;
- после сна:
  - двигает игровое время вперед;
  - отправляет событие `sleep_completed` в `PlayerMentalState`;
  - убирает rest-recoverable состояние `eye_strain`.

### 12.3. Формулы сна

Базовые коэффициенты сна:

| Параметр | Значение |
| --- | --- |
| Энергия в час | `12.5` |
| HP в час | `10` |
| Hunger в час | `-2` |
| Минимальный множитель восстановления энергии при 0 HP | `0.45` |

Полный reset сна считается от `6` часов.

### 12.4. Принудительный blackout sleep

Если время бодрствования достигает `48 часов`, `PlayerStats` запрашивает forced blackout.

В `game.gd` реализована отдельная последовательность:

- затемнение экрана;
- отметка старта сна;
- применение сна на `8 часов`;
- перерасчет статов через `SleepMath`;
- продвижение времени;
- событие `forced_blackout_sleep_completed` в ментальную систему;
- снятие `eye_strain`;
- возврат из затемнения.

### 12.5. Синхронизация сна с другими системами

Сон влияет на:

- скорость движения через общую энерго-усталостную модель;
- ментальное состояние;
- фриланс-условия `sleep_tired`, `sleep_very_tired`, `sleep_critical`;
- возможность аварийного blackout-сна.

## 13. Гигиена и состояние тела

Гигиена в проекте уже полноценная механика, а не просто декоративный параметр.

### 13.1. Пассивное падение гигиены

| Источник | Значение |
| --- | --- |
| Бодрствование | `1` в час |
| Сон | `0.5` в час |
| Один шаг | `0.03` |

### 13.2. Стадии гигиены

| Стадия | Диапазон |
| --- | --- |
| Clean | `76-100` |
| Untidy | `51-75` |
| Dirty | `26-50` |
| Unsanitary | `0-25` |

### 13.3. Эффекты стадий

`Dirty`:

- скорость движения `x0.95`;
- mood pressure `1.0`;
- social penalty weight `0.6`;
- illness risk `0.2`;
- alpha визуальной грязи `0.55`.

`Unsanitary`:

- скорость движения `x0.88`;
- mood pressure `2.0`;
- social penalty weight `1.0`;
- illness risk `0.45`;
- alpha визуальной грязи `0.90`.

`Untidy` тоже участвует в ментальных расчетах, но без тяжелой социальной реакции.

### 13.4. Источники грязи как задел под будущее

В `PlayerStats` уже заложены числовые значения для потенциальных источников загрязнения:

| Источник | Потеря гигиены |
| --- | --- |
| sex | `20` |
| fight | `14` |
| menstruation | `8` |
| body_fluids | `12` |
| dirty_work | `10` |
| blood | `10` |
| event_grime | `6` |
| humiliation | `8` |

Это важный маркер: система проектировалась с расчетом на более широкий набор событий, чем уже подключен в текущей версии.

### 13.5. `PlayerBodyState`

Отдельно от гигиены сохраняются:

- `body_dirt`;
- `body_blood`.

Особенности:

- видимая грязь в основном вычисляется из уровня гигиены;
- кровь хранится как независимый слой;
- кровь может очищаться отдельно;
- `wash_body(true)` смывает и грязь, и кровь.

### 13.6. Ванна

Ванна — отдельный interactable с modal dialog.

Характеристики:

- длительность процедуры: `45` игровых минут;
- после ванны:
  - гигиена восстанавливается;
  - тело отмывается;
  - кровь смывается;
  - отправляется mental event `bath_completed`;
  - показывается уведомление о результате.

Если гигиена реально выросла, показывается "полный" clean notification. Если игрок и так был почти чист, показывается более мягкий вариант.

### 13.7. Социальные комментарии NPC на грязь

`PlayerStats` умеет ограниченно выдавать комментарии NPC:

- комментарии появляются только при `dirty` и `unsanitary`;
- по одному NPC комментарий не повторяется бесконечно;
- можно комментировать максимум раз в день на NPC, если severity не выросла.

Сейчас это уже подключено к кассиру в магазине.

Если кассир делает замечание:

- запускается диалоговая последовательность;
- в `PlayerMentalState` отправляется событие `hygiene_humiliation_comment`;
- при `unsanitary` оно усиливается множителем `1.25`.

## 14. Ментальное состояние: настроение и стресс

`PlayerMentalState` — одна из самых важных систем, потому что она связывает быт, деньги, аренду, работу и социальное давление в единый контур игрового давления.

### 14.1. Базовые значения

| Параметр | Значение |
| --- | --- |
| Mood default | `62` |
| Stress default | `18` |
| Mood base | `60` |
| Stress base | `18` |
| Диапазон | `0..100` |

### 14.2. Естественный drift

Если mood ниже базы:

- восстанавливается `+0.45/час`.

Если mood выше базы:

- медленно спадает `-0.30/час`.

Если stress выше базы:

- снижается `-0.90/час`.

Если stress ниже базы:

- понемногу растет `+0.15/час`.

Если stress выше `55`, он начинает дополнительно давить на mood, до `-2.60/час` на верхних значениях.

### 14.3. Стадии mood

| Стадия | Порог | Влияние |
| --- | --- | --- |
| Excellent | `>= 78` | `sleep_recovery +0.12`, `work_energy_cost -0.08` |
| Normal | `>= 45` | без штрафов |
| Low | `>= 22` | `sleep_recovery -0.10`, `work_energy_cost +0.08` |
| Depressed | `< 22` | `sleep_recovery -0.22`, `work_energy_cost +0.18` |

### 14.4. Стадии stress

| Стадия | Порог | Влияние |
| --- | --- | --- |
| Calm | `< 24` | `sleep_recovery +0.06`, `work_energy_cost -0.05` |
| Tense | `< 49` | `sleep_recovery -0.04`, `work_energy_cost +0.05` |
| High | `< 74` | `sleep_recovery -0.12`, `work_energy_cost +0.12` |
| Panic | `>= 74` | `sleep_recovery -0.22`, `work_energy_cost +0.25` |

### 14.5. Контекстные модификаторы

Система автоматически подмешивает модификаторы от:

- голода;
- недосыпа;
- гигиены;
- `eye_strain`;
- нахождения дома;
- бедности;
- статуса аренды.

Примеры:

- домашняя безопасная зона: `mood +0.35/час`, `stress -0.60/час`;
- poverty:
  - если общие деньги `<= 120`: `mood -0.20`, `stress +0.35`;
  - если общие деньги `<= 35`: `mood -0.45`, `stress +0.75`;
- rent due:
  - `mood -0.35`, `stress +0.65`;
- rent overdue:
  - `mood -0.80`, `stress +1.25`.

### 14.6. Event-driven эффекты

Прямые события:

| Event | Mood | Stress | Дополнительно |
| --- | --- | --- | --- |
| `consume_food` | `+4` | `-3` | масштабируется полезностью еды |
| `bath_completed` | `+7` | `-10` | временный bath relief на `240` минут |
| `sleep_completed` | `+6` | `-8` | временный afterglow на `360` минут |
| `forced_blackout_sleep_completed` | `+3` | `-4` | мягче обычного сна |
| `freelance_success` | `+4` | `-1.5` | за успешную работу |
| `freelance_fail` | `-8` | `+10` | за провал |
| `rent_due_today` | `-4` | `+8` | в день оплаты |
| `rent_overdue` | `-8` | `+12` | при просрочке |
| `rent_paid` | `+6` | `-12` | после оплаты |
| `hygiene_humiliation_comment` | `-6` | `+9` | социальное унижение |

### 14.7. Финальные боевые модификаторы mental-системы

На выходе `PlayerMentalState.get_effects()` возвращает:

- `sleep_recovery_multiplier`, минимум `0.25`;
- `work_energy_cost_multiplier`, минимум `0.40`.

Они напрямую влияют на:

- эффективность сна;
- расход энергии в фрилансе.

## 15. Инвентарь, предметы и свежесть еды

### 15.1. Категории предметов

`ItemData` поддерживает:

- `misc`;
- `food`;
- `equipment`.

Для экипировки есть слоты:

- `top`;
- `bottom`;
- `shoes`;
- `head`.

### 15.2. Инвентарь игрока

`PlayerInventory`:

- размер по умолчанию: `20` слотов;
- хранит `InventorySlotData`;
- для еды учитывает свежесть;
- для экипировки создает уникальный `instance_id`;
- не дает удалить экипированный экземпляр без специального флага;
- не позволяет есть испорченное;
- умеет проверять, можно ли продать слот.

### 15.3. Продажа предметов

Система уже умеет определять продаваемость слота, но с жесткими условиями:

- предмет не должен быть экипирован;
- предмет должен иметь `can_sell = true`;
- предмет не должен быть `starter_locked`;
- `fixed_sell_price > 0`.

Технически выплата может идти:

- в наличные;
- в банк.

### 15.4. Холодильник

`FridgeInventory` — отдельное хранилище с собственной скоростью порчи:

- размер по умолчанию: `12` слотов;
- spoilage multiplier: `0.1`;
- может принимать заказ целиком только если весь набор влезает;
- умеет хранить, отдавать и позволять съесть еду прямо из холодильника.

### 15.5. Свежесть еды

`FoodFreshness` хранит состояние еды в минутах остатка срока.

Ключевые правила:

- игрок/доставка: множитель порчи `1.0`;
- холодильник: множитель порчи `0.1`;
- в один stack объединяются только продукты с совместимой свежестью;
- для еды сошедшей в ноль состояние считается `Испорчено`.

## 16. Экипировка и внешний вид

`PlayerEquipment` агрегирует все бонусы экипировки:

- stealth;
- attractiveness;

Финальный множитель движения:

На текущем наборе стартовой одежды заметных численных бонусов не задано, но сама система уже готова к расширению.

### 16.1. Визуальный пайплайн игрока

Визуал Руны собирается послойно:

- слой грязи;
- слой крови.

То есть грязь и кровь уже не просто влияют на цифры, а потенциально меняют внешний вид персонажа.

## 17. Экономика и деньги

`PlayerEconomy` хранит два кошелька:

- `cash` — наличные;
- `bank` — банковский счет.

### 17.1. Стартовые значения

- наличные: `$25`;
- банк: `$0`.

### 17.2. Что умеет экономика

- тратить наличные;
- тратить деньги с банка;
- класть наличные на счет;
- снимать деньги со счета;
- вести суточную сводку доходов/расходов;
- хранить историю по последним 7 дням.

### 17.3. Где используются наличные

Наличные сейчас нужны как минимум для:

- оплаты аренды;
- покупок у кассира в супермаркете;
- получения награды из `Slithario`;
- возможных будущих офлайн-трат.

### 17.4. Где используется банк

Банковый счет нужен для:

- получения оплаты за фриланс;
- покупок в desktop-магазине;
- отображения банковой истории;
- работы банкомата.

### 17.5. Банкомат

`ATMWindow` позволяет:

- снимать:
  - `$10`;
  - `$20`;
  - `$50`;
  - `$100`;
  - `$200`;
  - произвольную сумму;
- вносить те же суммы или свою;
- при внесении создавать запись в bank history через `FreelanceState`.

## 18. Аренда квартиры

`ApartmentRentState` — отдельная крупная подсистема.

### 18.1. Цикл аренды

- длина цикла: `7 дней`;
- первый платеж:
  - due через `6 дней` от момента инициализации;
  - сумма: `$1700`;
- следующие циклы:
  - сумма: `$700`.

### 18.2. Важные тайминги

- reminder накануне: `10:00`;
- день оплаты: `10:00`;
- первое сообщение о просрочке: на следующий день в `11:00`;
- follow-up при просрочке: каждые `3` дня в `11:00`.

### 18.3. Что хранит система

- текущий цикл аренды;
- статус `upcoming/due/overdue/paid`;
- landlord feed;
- unread count;
- calendar events;
- историю оплат.

### 18.4. Как платится аренда

`pay_current_rent()`:

- доступен только при `due` или `overdue`;
- берет деньги только из наличных;
- после оплаты:
  - помечает цикл оплаченным;
  - добавляет запись в историю;
  - добавляет сообщение от арендодателя;
  - закрывает текущие calendar events;
  - строит следующий цикл;
  - отправляет signal `rent_paid`.

### 18.5. Где это видно игроку

Аренда интегрирована сразу в несколько интерфейсов:

- `LeChatWindow`;
- desktop status widget;
- ментальная система;
- objective/pressure loop;
- календарные события.

## 19. Сюжет, цели и диалоги

### 19.1. `StoryState`

Сейчас хранит:

- primary objective;

### 19.2. Пролог

Пролог включается автоматически при первом входе в квартиру, если еще не был показан.

### 19.3. Диалоговая система

## 22. Магазины, доставка и хранение

### 22.1. Онлайн-магазин с компьютера

`ShopWindow` в desktop-режиме работает так:

- `payment_source = bank`;
- `use_delivery = true`;
- покупка не идет в инвентарь сразу;
- создается delivery на `4 игровых часа`.

### 22.2. Статусы доставки

`DeliveryManager` использует:

- `in_transit`;
- `awaiting_fridge_space`;
- delivered history.

Если в холодильнике нет места:

- заказ не пропадает;
- он ждет освобождения места.

### 22.3. Касса супермаркета

Касса работает иначе:

- `payment_source = cash`;
- `use_delivery = false`;
- товар сразу кладется в инвентарь игрока;
- перед открытием окна кассир может прокомментировать грязный вид Руны.

### 22.4. Важная техническая особенность кассы

В `cashier_shop_interactable.gd` поведение уже верное: товар идет сразу в инвентарь. Но в `Cashier.tscn` поле `window_subtitle_text` все еще содержит устаревший текст про "доставку через 4 игровых часа". По факту доставка у кассы не используется.

Это стоит учитывать как текущую рассинхронизацию текста и реальной логики.

### 22.5. Окно хранения

`StorageWindow` поддерживает:

- левую колонку инвентаря игрока;
- правую колонку хранилища;
- выбор строки;

  - `Положить`;

  - `Закрыть`;

- отображение свежести предметов.

## 23. Фриланс

`FreelanceState` — основной карьерный и денежный цикл игры.

### 23.1. Режим рабочего дня

- начало работы: `06:00`;
- закрытие рабочего дня: `22:00`;
- после начала дня генерируется `2 заказа`.

### 23.2. Базовые параметры фриланса

| Параметр | Значение |
| --- | --- |
| Стартовый service rating | `50` |
| Максимальный rating | `100` |
| Стартовый moderation level | `1` |
| Максимальный level | `15` |
| Стартовый XP | `0` |
| Длительность заказа | `90` игровых минут |
| Шанс urgent | `15%` |
| Шанс urgent при rating >= 75 | `25%` |
| Множитель награды urgent | `1.35` |

### 23.3. Сложность заказов

Количество комментариев на заказ:

| Сложность | Комментариев |
| --- | --- |
| Easy | `7` |
| Medium | `12` |
| Hard | `17` |

Базовая награда:

| Сложность | База |
| --- | --- |
| Easy | `$40` |
| Medium | `$65` |
| Hard | `$90` |

Выбор сложности зависит от rating: чем он выше, тем меньше easy-заказов и тем выше доля medium/hard.

### 23.4. Энергозатраты работы

Базовые значения:

- успешный заказ: `-15 energy`;
- провал: `-40 energy`.

Оба значения затем домножаются на `PlayerMentalState.work_energy_cost_multiplier`.

### 23.5. Оценка результата

| Accuracy | Результат |
| --- | --- |
| `< 0.60` | fail |
| `0.60..0.89` | normal |
| `>= 0.90` | excellent |

### 23.6. Награда, рейтинг и XP

Награда:

- excellent: `1.25x base`;
- normal: `1.0x base`;
- fail: `0`.

Изменение рейтинга:

- excellent: `+5`;
- normal: `+1`;
- fail urgent: `-10`;
- fail non-urgent: `-7`.

XP:

| Сложность | Excellent | Normal |
| --- | --- | --- |
| Hard | `18` | `12` |
| Medium | `15` | `10` |
| Easy | `12` | `8` |

Порог XP на уровень:

- `25 + (level - 1) * 15`

### 23.7. Выплата

Фриланс платит не наличными, а на банк:

- при завершении заказа деньги идут в `PlayerEconomy.add_bank_dollars(...)`;
- создается запись в банковой истории;
- появляется bank notification.

### 23.8. Eye strain

После `2` и более завершенных заказов за день добавляется условие `eye_strain`.

Оно:

- ухудшает mental-состояние;
- влияет на мини-игру модерации визуальными помехами;
- снимается отдыхом/сном.

### 23.9. Условия из других систем

`FreelanceState` — не просто работа, а центральный реестр активных conditions. В него синхронизируются:

- hunger-условия;
- sleep-условия;
- hygiene-условия;
- mental-условия.

Это позволяет в UI фриланса показывать, почему игрок сейчас работает хуже.

## 24. Мини-игра модерации

`ModerationMinigameUI` — геймплейное ядро фриланса.

### 24.1. Вход в мини-игру

- игрок выбирает заказ в `FreelanceAppWindow`;
- desktop открывает `ModerationMinigameUI`;
- мини-игра стартует по `order_id`.

### 24.2. Тайминги

| Сложность | Время на комментарий |
| --- | --- |
| Easy | `3.2` сек |
| Medium | `2.4` сек |
| Hard | `1.7` сек |

Urgent-заказ уменьшает время множителем `0.8`.

### 24.3. Управление

- `A` / `Left` = approve;
- `D` / `Right` = reject.

### 24.4. Логика раунда

- пул комментариев загружается из JSON;
- при необходимости используются fallback comments;
- сессия старается держать разнообразие safe/reject-комментариев;
- timeout считается неправильным действием;
- accuracy считается как `correct / total`.

### 24.5. Eye strain в мини-игре

Если у игрока есть `eye_strain`, включаются дополнительные визуальные помехи:

- blink interval: `2.6..5.0` сек;
- blink duration: `0.06..0.12` сек;
- blink alpha: `0.78`.

Это делает состояние не только числовым штрафом, но и реальной геймплейной помехой.

## 27. Сохранения и загрузка

### 27.1. Формат

`SaveManager` использует:

- autosave: `user://saves/autosave.json`;
- manual slots: `slot_01.json` ... `slot_10.json`.

### 27.2. Автосейв

Автосохранение:

- работает каждые `15` реальных минут;
- включается только когда активна `game.tscn`.

### 27.3. Что сохраняется

Сохраняются snapshot-ы основных глобальных систем:

- статы;
- инвентарь;
- экипировка;
- экономика;
- время;
- холодильник;
- доставки;
- фриланс;
- аренда;
- ментальное состояние;
- сюжет;
- настройки room/runtime состояния.

Отдельно хранится summary:

- комната;
- день/время;
- наличные;

### 27.4. Загрузка

При загрузке:

- runtime UI закрывается;
- глобальные состояния ресетятся;
- payload автозагрузок применяется;
- загружается `game.tscn`;
- затем через `GameManager` восстанавливается мир и игрок.

## 29. Основной игровой цикл в терминах механик

На уровне дизайна проект уже складывается в замкнутый loop:

1. Игрок просыпается и живет в квартире.
2. Время идет, голод и грязь растут, настроение и стресс меняются.
3. Нужно:
   - поесть;
   - помыться;
   - следить за энергией;
   - не довести себя до blackout.
4. Деньги добываются:
   - через фриланс;
   - через `Slithario`.
5. Еду можно:
   - купить онлайн дороже, но с банка;
   - купить в супермаркете дешевле, но за наличные.
6. Заказы и бедность давят на mental-систему.
7. При приближении аренды растет стресс.
8. Игрок должен свести воедино:
   - бытовое выживание;
   - доход;
   - логистику еды;
   - поддержание рабочего состояния;
   - оплату аренды.

Это уже не набор несвязанных систем, а вполне сформированный цикл выживания и бытового менеджмента.
